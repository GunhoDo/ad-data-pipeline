# Data Lake 1세대 기본 구조 
>Spark SQL → Glue/HMS → S3
### 장점 : 저비용, 유연, 멀티 엔진
### 단점 : 데이터가 커지면 문제가 됨, 모든 문제의 뿌리는 하나
>“카탈로그는 폴더 위치만 알지, 파일 안에는 뭐가 있는지 모른다”
* **파티션 리스팅 지옥**, 일상적 고통, 쿼리가 느려짐 N+1 쿼리 문제 파티션 하나당 쿼리를 날리는 개별 조회 + 통합 조회
* **파티셔닝 변경 불가**, 책장을 장르별로 정리했는데 출판 연도별로 바꾸려면 책을 전부 꺼내서 다시 꽂아야 하는 상황 근본 원인은 물리적 디렉토리 구조 = 논리적 파티션 구조 둘이 결합되어 있어 떼어낼 수 없음
* **ACID** 없음, 동시 쓰기 상황에서의 데이터 유실, 지연 이벤트 재처리, 장애 복구처럼 쓰기가 겹치는 순간, 치명적 장애
* **롤백 불가**, 스냅샷이 없어 복구 불가능, S3 Versioning, 수동 백업 Glacier 아카이빙이 있으나 모두 비용 + 수동 복구 프로세스, 테이블 단위가 아닌 파일 단위로 타임 스탬프 보면서 하나씩 백업, 복구 동안 서비스 전체가 반나절 멈춤
